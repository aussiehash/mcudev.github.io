TREZOR Dev Kit

Introduction

This hobbyist project documents a kit that developers can use to explore, develop, test, and debug the open source TREZOR hardware and software.

Warning

This kit does NOT produce a production version of the TREZOR cryptocurrency hardware wallet.

If you want an official TREZOR hardware wallet, see: https://shop.trezor.io/

When contrasted to official production hardware, this kit lacks key security properties that make it less safe to use for non-development purposes. For example, when used as documented here, this kit purposefully does not lock the flash memory that stores the bootloader code. This is so that developers may repeatedly develop and test bootloader code changes. Production hardware has these memory areas locked so that they can not be changed by an attacker. There are other subtle differences as well, that may matter, depending on your usage and threat model.

About this Kit

One key difference between this kit and production hardware is the MCU (micro-controller) used. Production hardware uses the STM32F205RET6 MCU and this kit uses the STM32F405RGT6 MCU. The former is a Cortex-M3 and the latter is a Cortex-M4. The Cortex-M4 architecture is a backwards compatible superset of Cortex-M3. The additional features, like hard float capability, do not matter for this project. The STM32F405RGT6 has 1MB of flash memory (vs the 205's 512KB), a core clock frequency of 168MHz (vs the 205's 120MHz), and 192KB of SRAM (vs the 205's 128KB). The STM32F405RGT6 is pin-to-pin compatible with the STM32F205RET6 and uses the same TRNG.

The Waveshare Core405R dev board has an 8MHz high-speed external crystal (matching the reference hardware), a STM32F405RGT6 MCU, a USB connector, power circuitry, SWD debug interface, boot mode select switch, and all the pins that we need, broken out and available for use.

Lastly, the STM32F405RGT6 is used by the MicroPython pyboard v1.1 and is being used to develop the TREZOR v2.

Setup

This setup process comprises steps to install development and debugging tools on a temporary computer, as well as building and installing an initial version of the bootloader and firmware on the kit's MCU.

The following setup process was documented while running a Ubuntu GNOME 17.04 Live CD (Zesty Zapus) as user ubuntu-gnome.

This process has been documented in this video, if you'd like to follow along.

  1. Connect to the Internet
  2. Install required software packages
    • sudo apt-get update
    • sudo apt-get -y install git
  3. Download tools
    GNU ARM Embedded Toolchain
    OpenOCD (see also http://openocd.org/doc-release/pdf/openocd.pdf)
    • cd /tmp/
    • wget https://developer.arm.com/-/media/Files/downloads/gnu-rm/6-2017q2/gcc-arm-none-eabi-6-2017-q2-update-linux.tar.bz2
    • wget https://github.com/gnu-mcu-eclipse/openocd/releases/download/gae-0.10.0-20170418/gnuarmeclipse-openocd-debian64-0.10.0-201704182147-dev.tgz
  4. Extract tools
    • tar xjf gcc-arm-none-eabi-6-2017-q2-update-linux.tar.bz2
    • tar xzf gnuarmeclipse-openocd-debian64-0.10.0-201704182147-dev.tgz
  5. Add extracted tool binaries to PATH environment variable
    • export PATH="/tmp/gcc-arm-none-eabi-6-2017-q2-update/bin:/tmp/gnuarmeclipse/openocd/0.10.0-201704182147-dev/bin:$PATH"
  6. Make the USB devices that we will be using available to non-root users
    • /bin/echo -e '# debug adapter\nSUBSYSTEMS=="usb", ATTRS{idVendor}=="0483", ATTRS{idProduct}=="3748", MODE:="0666"' > /tmp/99-dev-kit.rules
    • /bin/echo -e '# dev board running trezor v1\nSUBSYSTEMS=="usb", ATTRS{idVendor}=="534c", ATTRS{idProduct}=="0001", MODE:="0666"' >> /tmp/99-dev-kit.rules
    • sudo cp /tmp/99-dev-kit.rules /etc/udev/rules.d/
  7. Download the TREZOR v1 bootloader and firmware source code
    • git clone https://github.com/trezor/trezor-mcu.git
  8. Build the TREZOR v1 bootloader and package it with a stock 1.5.0 firmware
    Note: The "export MEMORY_PROTECT=0" is what controls the code that locks the bootloader's flash memory on your dev board.
    • cd trezor-mcu/
    • git submodule update --init
    • CFLAGS='-std=c99' make -C vendor/libopencm3/
    • export OPTFLAGS=-Os
    • export MEMORY_PROTECT=0
    • make
    • make -C bootloader/
    • ./bootloader/firmware_align.py bootloader/bootloader.bin
    • cp bootloader/bootloader.bin bootloader/combine/bl.bin
    • cd bootloader/combine/
    • wget -O fw.bin https://wallet.trezor.io/data/firmware/trezor-1.5.0.bin
    • ./prepare.py
  9. Begin the process to put the combined bootloader and firmware code onto the dev board
    First, plug-in both the dev board and the debug adapter to your computer's USB ports.
    • cd /tmp/gnuarmeclipse/openocd/0.10.0-201704182147-dev && ./bin/openocd -f ./scripts/interface/stlink-v2.cfg -f ./scripts/target/stm32f4x.cfg
    • In a second terminal, start gdb and connect it to the openocd that we started in the previous step with:
      /tmp/gcc-arm-none-eabi-6-2017-q2-update/bin/arm-none-eabi-gdb /tmp/trezor-mcu/bootloader/bootloader.elf -ex 'target remote localhost:3333'
    • Within gdb, pass the control and flash commands through to openocd:
      (gdb) monitor reset halt
    • (gdb) monitor flash write_image erase /tmp/trezor-mcu/bootloader/combine/combined.bin 0x08000000
    • At this point, the flash has been written. Now set some test breakpoints and simulate debugging some code.
      (gdb) monitor reset halt
    • (gdb) b main
    • (gdb) c
    • (gdb) b memory_protect
    • (gdb) c
    • (gdb) c
    • (gdb) Press Ctrl + C to interrupt software execution
    • (gdb) quit
    • Done flashing the software to the dev board. Unplug the debug adapter and dev board.
  10. Test the bootloader and firmware just loaded onto the dev board
    First, plug-in the dev board
    • cd /tmp/
    • git clone https://github.com/trezor/python-trezor.git
    • sudo apt-get -y install python-pip python-dev cython libusb-1.0-0-dev libudev-dev
    • sudo pip install setuptools click trezor
    • ./python-trezor/trezorctl wipe_device
    • ./python-trezor/trezorctl load_device -m 'fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun fun agent'
    • ./python-trezor/trezorctl get_address -c Bitcoin -n "Bitcoin/0'/0/0"
    • ./python-trezor/trezorctl get_address -c Litecoin -n "Litecoin/0'/0/0"
    • ./python-trezor/trezorctl get_address -c Namecoin -n "Namecoin/0'/0/0"
  11. Install eclipse so that we can debug in a graphical environment
    • cd /tmp/
    • sudo apt-get -y install openjdk-8-jdk
    • wget http://mirror.math.princeton.edu/pub/eclipse//technology/epp/downloads/release/oxygen/R/eclipse-cpp-oxygen-R-linux-gtk-x86_64.tar.gz
    • tar xzf eclipse-cpp-oxygen-R-linux-gtk-x86_64.tar.gz
    • mkdir -p /tmp/eclipse-workspace
  12. Start OpenOCD and eclipse. Setup a debug configuration and debug graphically.
    The full process is shown in the video in Figure 16.

More

Running Standalone

The kit can also be run in a standalone configuration, without the debug adapter connected.

Flashing in DFU mode with dfu-util

The preceding process to flash code into the MCU's flash memory was done in non-Device Firmware Upgrade (DFU) mode. DFU mode can be used also. Doing so just requires using different tools, moving a switch, and unplugging and replugging the dev board a couple of times. Using DFU mode is a little less convenient when compared to the flash process used above.

To use DFU mode

  • Unplug the dev board and debug adapter if either is plugged into your computer's USB port(s)
  • Flip the Boot Config switch to System/DFU mode (Number 2 in Figure 8 above)
  • Plug the dev board into your computer's USB port
  • Verify with:
    lsusb | grep 'STMicroelectronics STM Device in DFU Mode'
    You'll see something like:
    0483:df11 STMicroelectronics STM Device in DFU Mode
  • cd /tmp/
  • sudo apt-get -y install dfu-util
  • dfu-util -a 0 -s 0x8000000 -D /tmp/trezor-mcu/bootloader/combine/combined.bin
  • Wait until finished and then unplug the dev board
  • Flip the Boot Config switch back to Flash mode (Number 2 in Figure 8 above)
  • Plug the dev board into your computer's USB port. You're now running the code that you just stored into the MCU's flash memory.

Getting Info About and Erasing MCU Flash

Here are some more gdb-to-openocd passthrough commands for working with the MCU flash memory.

  • Retrieve info about current state of the flash memory sectors
    (gdb) monitor flash info 0
  • Erase all flash memory sectors
    (gdb) monitor flash erase_sector 0 0 last
